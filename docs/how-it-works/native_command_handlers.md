# Native Command Handlers: Profile-Aware Honeypot Responses

## Overview

Project Violet implements a **two-tier hybrid command execution system** inside the Cowrie honeypot:

1. **Native Command Handlers** (deterministic, profile-aware) — handle common commands with static output derived from the deployed profile
2. **LLM Fallback** (dynamic) — routes unsupported commands to a language model for realistic interactive responses

Native handlers are compiled from the profile during deployment via `profile_converter.py` and stored across four artifact types:

| Artifact | Format | Used By |
|----------|--------|---------|
| `fs.pickle` | Binary-serialized filesystem tree | `ls`, `cat`, `head`, `tail`, `find`, `grep`, `du` |
| `honeyfs/` | Real files on disk | `cat`, `head`, `tail`, `grep` (file contents) |
| `txtcmds/` | Static text files | `uname`, `hostname`, `df`, `free`, `netstat`, `last` |
| `cmdoutput.json` | JSON structure | `ps` (process list) |

## Handler Categories

### A. Process and Service Management (Highly Profile-Aware)

#### `ps` (`cowrie/commands/base.py`)

Reads from **two sources**:

1. **`cmdoutput.json`** (preferred) — generated by `profile_converter.py.generate_cmdoutput()`, stores a structured process list with USER, PID, CPU%, MEM%, VSZ, RSS, TTY, STAT, START, TIME, COMMAND per entry.
2. **Hardcoded fallback** — kernel processes, getty, rsyslogd, SSH if `cmdoutput.json` is unavailable.

The profile generator computes realistic VSZ/RSS values per service: `apache2` gets (171680, 5244), `mysqld` gets (1793204, 178432), `dockerd` gets (1564832, 89424), etc. Supports `-a`, `-x`, `-u`, `-w` flags with dynamic column width calculation.

#### `systemctl` (`cowrie/commands/systemctl.py`)

Reads `profile["services"]` via the LLM fallback handler's `_profile` attribute. Each service has `name`, `pid`, `command`, `user`, `ports`.

- `status <service>` — three-level name matching (exact, loose strip-suffix, substring)
- `list-units` — enumerates all profile services plus standard system units
- `is-active` / `is-enabled` — checks existence in profile; all profile services report as "active" and "enabled"

Generates realistic systemd output with cgroup paths and boot timestamps.

#### `ss` (`cowrie/commands/ss.py`)

Reads `profile["services"]` to generate socket statistics. For each service port, generates a TCP LISTEN entry. Resolves numeric port names (22 → "ssh", 3306 → "mysql") via an internal `_PORT_NAMES` dict.

Supports `-t` (TCP), `-u` (UDP), `-l` (listen), `-n` (numeric), `-p` (show PID/process), `-a` (all). Shows the current SSH connection as an ESTABLISHED entry using `self.protocol.kippoIP` and `self.protocol.clientIP`.

#### `journalctl` (`cowrie/commands/journalctl.py`)

Reads `profile["services"]` for service matching, then looks up log files via `_SERVICE_LOG_MAP` (service → log path mapping). Attempts to read actual log content from honeyfs; falls back to per-service log templates (`_SERVICE_TEMPLATES`) with realistic systemd journal timestamps.

Supports `-u <unit>` (filter by service), `-f` (follow), `-n` (lines), `-xe` (extended).

### B. Filesystem Operations (Profile-Aware via fs.pickle)

All filesystem commands read from `self.protocol.fs` — a `HoneyPotFilesystem` initialized from the profile-generated `fs.pickle`.

#### `ls` (`cowrie/commands/ls.py`)

Every file and directory comes from the profile's `directory_tree`. Respects file permissions, ownership, timestamps, and sizes. Supports `-l` (detailed), `-h` (human-readable sizes), `-a` (hidden files), and calculates "total blocks" for realistic output.

#### `cat` (`cowrie/commands/cat.py`)

Resolves paths against the pickled filesystem, then reads file content bytes from honeyfs or pickle-embedded data. Supports `-n` (line numbers), multiple files, and reports "Is a directory" for directory paths.

#### `head` / `tail` (`cowrie/commands/head.py`)

Same filesystem access as `cat`. `head` outputs first N lines (default 10), `tail` outputs last N lines. Both support `-n` for line count. `tail -f` simulates follow mode with continuous output. Multiple files show `==> filename <==` headers.

#### `find` (`cowrie/commands/find.py`)

Recursive filesystem traversal from `fs.pickle`. Supports `-name` (glob pattern via `fnmatch`), `-type` (f/d), `-maxdepth`. Unsupported flags (e.g., `-newer`, `-perm`) route to LLM fallback if `hybrid_llm` is enabled.

#### `grep` (`cowrie/commands/grep.py`)

Searches file contents from honeyfs and the pickle filesystem. Profile-aware: results reflect actual file contents planted by the profile.

#### `du` (`cowrie/commands/du.py`)

Calculates directory sizes from the pickle filesystem's stored file sizes. Supports `-sh` (summary, human-readable).

### C. Package Management (Profile-Aware)

#### `dpkg` (`cowrie/commands/dpkg.py`)

Reads `profile["installed_packages"]` — array of `{name, version, architecture}` dicts.

- `-l` — lists all packages with version numbers in Debian dpkg format
- `-s <pkg>` — detailed status for a requested package (if in profile)
- `-L <pkg>` — generates plausible file ownership lists (auto-generated from package name: `/usr/bin/{pkg}`, `/usr/share/doc/{pkg}`, etc.)

### D. Container Inspection (Profile-Aware)

#### `docker` (`cowrie/commands/docker.py`)

Extracts container names from docker-compose files in `profile["file_contents"]`, falls back to service names from `profile["services"]`.

- `ps` — lists containers with deterministic IDs (MD5 hash of name)
- `images` — shows image repos/tags derived from services
- `inspect` — returns JSON with fake container config
- `logs` — tries honeyfs log files first, then generates generic startup messages
- Swarm commands (`docker node ls`, `docker service ls`) supported
- Non-read commands (`exec`, `run`, `build`) trigger LLM fallback

### E. User Information (Profile-Aware)

#### `id` (`cowrie/commands/base.py`)

Reads `profile["users"]` to find the logged-in user's groups. Maps group names to GIDs via `_KNOWN_GIDS` dict (root=0, adm=4, sudo=27, docker=999, etc.) or generates auto-incrementing GIDs for unknown groups.

Output: `uid=1000(alice) gid=1000(alice) groups=1000(alice),4(adm),24(cdrom)`

### F. Static Command Outputs (txtcmds)

Generated by `profile_converter.py.generate_txtcmds()` and written as flat files. Cowrie reads these before checking for native handlers.

| Command | Source |
|---------|--------|
| `uname -a` | `profile["system"]` (hostname, kernel, arch) |
| `hostname` | `profile["system"]["hostname"]` |
| `whoami` | Always "root" |
| `uptime` | Hardcoded "47 days up" |
| `ps aux` | Profile services with realistic VSZ/RSS |
| `ifconfig` | `profile["network"]` interfaces |
| `netstat -tlnp` | Profile services and ports |
| `df -h` | `profile["disk_layout"]` |
| `free -m` | Hardcoded realistic memory values |
| `last` | Generated login history spanning ~90 days |

## Profile Access Pattern

All profile-aware handlers follow the same pattern:

```python
handler = getattr(self.protocol, "llm_fallback_handler", None)
if handler is None:
    return []  # Graceful fallback to empty/defaults
profile = getattr(handler, "_profile", {})
data = profile.get("desired_key", [])
```

## LLM Fallback Integration

Commands that don't fully handle all flags fall through to the LLM:

```python
if unsupported_flag and hybrid_llm_enabled and handler:
    d = handler.handle_command(original_cmd_string)
    d.addCallback(self._write_and_exit)
    return
```

## Cross-Command Consistency

All commands that reference the same entity produce consistent information because they read from the same profile. If `profile["services"]` lists nginx on port 80, then:

- `ps` shows an nginx process
- `systemctl status nginx` shows it running
- `ss -tlnp` shows port 80 listening
- `docker ps` shows an nginx container (if docker-compose present)
- `journalctl -u nginx` shows nginx logs

This consistency is critical for fooling both human and LLM attackers.
